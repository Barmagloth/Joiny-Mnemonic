# HANDOFF2 — задача: аудит механизмов переходов состояний

Самодостаточная задача для новой сессии. Дата постановки: 2026-07-18.
Постановщик: пользователь; черновой сбор фактов начат, оборван сознательно.

## Вопрос

В кодовой базе несколько подсистем ведут жизненные циклы записей.
Требуется установить: **это разные автоматы, потому что предметные
области действительно разные, — или независимо написанные реализации
одного шаблона, разошедшиеся в деталях?** Итог — таблица по подсистемам
с осями (примерно: Type / Provenance / Trust / Lifecycle-flow /
Storage-shape) и вердикт по дублированию с указанием, где сходство
только внешнее. Формат таблицы из постановки — ТОЛЬКО ПРИМЕР; оси и
строки вывести из реального кода.

## Контекст проекта (минимум)

Joiny-Mnemonic — provenance-first память для LLM-агентов: append-only
журнал событий (SQLite, hash-chain), поверх — производные записи с
провенансом. Весь код переходов в `src/joiny_mnemonic/storage.py`.
Дисциплина проекта: каждое утверждение отчёта обязано указывать на
код/строку; «доказано/не доказано» разделять; никаких оценок «скорее
всего» без измерения (на этой неделе было 5 инцидентов подмены фактов
нарративом — см. handoff.md, раздел про переквалификацию гейта).

## Уже проверенные факты (с указателями, перепроверь при сомнении)

Механизмы переходов/циклов, найденные к моменту остановки:

1. **transition_candidate** (storage.py ~4143) над таблицей
   `candidate_transitions` — словарь extraction-кандидатов:
   auto/quarantined/confirmed/rejected/superseded/\*_requested.
   Провенанс ✓ (source event проверяется), траст ✓ (boundary-статусы
   требуют trusted origin, множество из 4 origins), **flow-матрицы НЕТ**
   (легальность from→to не проверяется), идемпотентности нет.
2. **settle_candidate** (storage.py ~4292) — ВТОРАЯ реализация над ТОЙ
   ЖЕ таблицей, словарь settlement: pending/applied/reverted/contested.
   Провенанс ✓, траст ✓ (ДРУГОЕ множество origins: + local_operator,
   delegated_agent, + policy-гейт агента), **flow ✓** (_SETTLEMENT_FLOW,
   терминалы, идемпотентные повторы). Самая строгая.
3. **transition_finding** (storage.py ~3670) над `finding_transitions` —
   security findings: active→acknowledgement_requested→acknowledged.
   Схема строки почти копия candidate_transitions (from/to/source/actor/
   origin, но БЕЗ rule_id). Провенанс ✓, траст ✓ (acknowledged требует
   trusted origin, то же множество из 4), **flow-матрицы НЕТ**.
4. **create_task_version** (storage.py ~3294) + tasks.py set_status —
   task_versions: enum {active,blocked,completed,cancelled} ✓, версии,
   провенанс ✓ (source_event_ids + snapshot). **Траста НЕТ** (set_status
   доступен из MCP/API без origin-гейта), **flow НЕТ** (любой→любой,
   cancelled→active пройдёт молча).
5. **block_versions / memory_records** — не статусные автоматы, а цепочки
   supersession (supersedes_id); отдельный паттерн версионирования.
6. **policy_ledger / activate_policy** (storage.py ~3702) — версионная
   цепочка с operation {activated,replaced,rollback}, траст ✓ (то же
   множество 4 origins), провенанс ✓.
7. **extraction_attempts** — outcome-словарь воркера (succeeded /
   retryable_failure / ... — вычитать полный словарь из кода ~3850).
8. **snapshots + snapshot_prunings** — created→pruned через канониче-
   ское событие; защищённые множества.

## Замеченные кандидаты в находки (проверить и оформить)

- **Три независимые реализации шаблона «аудируемый переход»** (1,2,3):
  одинаковая форма строки, разошедшиеся поведения — flow есть только у
  settlement; трастовые множества различаются; rule_id есть не везде.
  Вопрос: сознательная специализация или дрейф копипасты?
- **Несогласованность траста задач**: tasks.set_status меняет статус
  задачи (в т.ч. completed) без какого-либо origin-гейта, в то время
  как settlement для task_closure требует evidence + trusted origins.
  Два пути «завершить задачу» с разной строгостью — открытый вопрос
  task6.md («Should settlement tie into TaskManager?») это предвидел.
- Проверить: возможны ли нелегальные переходы на практике через
  публичные поверхности (rejected→confirmed у кандидата? повторное
  acknowledged у finding? cancelled→active у задачи?) — и написать
  падающие/фиксирующие тесты на найденное.

## Ожидаемый результат

1. Таблица «подсистема × оси» с ✓/частично/✗ и указателями на код.
2. Вердикт: где дублирование реальное (один шаблон, N реализаций,

4. Если предложишь унификацию — только эскиз с ценой и потребителем;
   ничего не реализовывать без явного «да».
